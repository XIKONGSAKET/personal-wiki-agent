# Product Roadmap Profile

For PMs and product teams prioritizing features, planning GTM, and structuring quarterly/annual roadmaps.

## When to use

- Quarterly or annual roadmap planning
- Feature prioritization across a backlog
- GTM strategy for a launch
- Reframing a stalled product direction
- Resource allocation across initiatives

## Phase customizations

### Phase 1 — Understanding
**Scoping questions:**
- What's the product's North Star Metric?
- What's the current quarter's primary objective?
- What's the engineering / design / data capacity?
- Who are the primary user segments?
- What recent data points (user research, churn, feature usage) are most relevant?

### Phase 2 — Context
**Tier 3 search topics:**
- Direct competitor recent launches and roadmap signals
- Adjacent product categories with similar problems
- Industry benchmarks on the metrics that matter
- Platform / API changes that constrain feature options

### Phase 3 — Generation
**Variant axes specific to product roadmap:**
- Build / buy / partner per feature
- Depth vs. breadth (one big feature vs. many small)
- Acquisition-focused vs. retention-focused vs. monetization-focused
- New segment expansion vs. existing segment deepening
- Platform expansion vs. core product investment

### Phase 4 — Deep Dive
**Specification must include:**
- Prioritized feature list with ICE or RICE scores
- Sequencing rationale (why this order)
- Cross-team dependencies
- Risk of each top-3 feature failing to ship in scope
- Success metric per feature, measurable within one quarter

### Phase 5 — Critique
**Premortem questions specific to product roadmap:**
- Which feature is most likely to ship late or with cut scope?
- What if the North Star Metric doesn't move despite shipping everything?
- What if user research turns out to be unrepresentative?
- What if a key engineering hire / departure changes capacity?
- What if a competitor ships a similar feature first?

### Phase 6 — Plan
**Planner deliverables specific to product roadmap:**
- Quarter-by-quarter sequencing
- Resource allocation per initiative (eng / design / data weeks)
- Kill criteria per major bet (when to deprecate)
- Stakeholder communication cadence
- Definition of "done" for each top-3 feature

## Frameworks heavily used

- ICE / RICE
- Kano model (basic / linear / delighter / indifferent)
- North Star Metric and input metrics
- MoSCoW prioritization
- Opportunity-Solution tree
- Jobs-to-be-Done
- Sequencing / Beachhead strategy
- Second-order effects analysis

## Anti-patterns specific to product roadmap

- Roadmap as a wishlist (everything is P1)
- Ignoring opportunity cost ("if we do X, we don't do Y")
- Pressing features without clear success metrics
- Letting sponsorship/loud-voice biases override data
- Ignoring technical debt as a hidden tax on velocity
- Cargo-culting features from competitors without user evidence

## Critical risks to surface in critique

1. North Star Metric won't move because top features address wrong constraint
2. Engineering capacity overestimated, scope cuts cascade
3. User research sample-biased — direction is wrong but feels validated
4. Cross-team dependency stalls the whole quarter
5. Competitor leapfrog makes shipped feature redundant
